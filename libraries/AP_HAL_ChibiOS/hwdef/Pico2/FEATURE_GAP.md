# Pico2 (RP2350) vs CubeBlack (STM32F427) — Feature Gap Analysis

**Branch:** `buzz-rp2350-chibios-v3`  
**Date:** 2026-03-30 (last updated)  
**Purpose:** Track what is implemented, what is partially working, and what still needs work on the Pico2 RP2350 port.  
MOST SOFTWARE IMPLEMENTATION IS COMPLETE, its at a satisfactory level, we are focusing on HARDWARE VERIFICATION now, with openocd , etc, moving forward.

**current hardware status (updated 2026-03-30, session 15) — ALL THREE PIOUART INSTANCES (SERIAL3/4/5) END-TO-END LOOPBACK VERIFIED ON HARDWARE:** With all three loopback jumpers installed (GP14→GP17, GP19→GP20, GP21→GP27), a simultaneous 40-round SERIAL_CONTROL test on `/dev/ttyACM1` returned SERIAL3=90%, SERIAL4=88%, SERIAL5=88% with zero bad-data replies across all 120 rounds. The no-reply misses are timing artefacts from three ports competing for the same 400 ms reply window, not byte corruption. Root cause of prior SERIAL5 corruption was an experimental PIO RX opcode (`0xEA37` = `set x,7 [18]`) left in `PIOUART.h`; reverted to the same midpoint-sampling opcode that works for SERIAL3/4 (`0xEA27` = `set x,7 [10]`). All three PIOUART instances are now end-to-end functional on live hardware.

**current hardware status (updated 2026-04-01, session 16) — RC INPUT IRQ21 STARTUP HARDFAULT RESOLVED ON HARDWARE:** The remaining RP2350 RC-in crash was no longer inside the original PAL callback path. Root cause turned out to be a chain of early-startup interactions: GPIO edge capture on GPIO16 was armed too early, CRSF telemetry singleton allocation could be triggered from the RCIN thread before full system init, the Pico2 hwdef was still using the default `MAIN_STACK=0x600`, and `hrt_micros64()` in RAM was compiled into a spill/reload sequence around `chSysUnlock()` that faulted under interrupt pressure. Fixes now in tree: (1) `SoftSigReaderRP2350` keeps GPIO16 configured early but defers `palEnableLineEventI()` until the RCIN timer path first runs; (2) `AP_RCProtocol_CRSF::bind_in_progress()` now refuses to instantiate telemetry before `hal.scheduler->is_system_initialized()`; (3) Pico2 `hwdef.dat` now sets `MAIN_STACK 0x1000`; (4) `hrt_micros64()` now uses inline `port_lock()` / `port_unlock()` primitives, eliminating the faulting `ldrd [sp]` sequence. **Verified 2026-04-01 on hardware:** after 15 s runtime, SWD halt landed in normal main-loop execution (`AP_DAL_InertialSensor::start_frame`), with `CFSR=0`, `HFSR=0`, `MSP=0x20001000`; a follow-up RCInput probe showed `_num_channels=8`, valid pulse widths (`ch0-3 = 1509,1497,1001,1496` µs), and the deferred GPIO IRQ path armed (`_irq_enabled=true`). Conclusion: the RC-in hardfault is resolved and RC pulse capture remains functional on live hardware.

**current hardware status (updated 2026-03-30, session 11) — PIOUART SERIAL3/SERIAL4 END-TO-END LOOPBACK NOW WORKING ON HARDWARE:** Final root cause for the last SERIAL3 contamination was higher-level ownership rather than the PIO state machines: direct `SERIAL_CONTROL` access to `SERIAL3/SERIAL4` now also locks GPS consumers when `EXCLUSIVE` is requested, so GPS probe/driver traffic cannot pollute loopback validation. On the live board, short blocking `SERIAL_CONTROL` tests on `/dev/ttyACM1` now return clean echoed payloads from both PIO ports: `device=103` repeatedly returns `3131...` (16 x `'1'`) and `device=104` repeatedly returns `6262...` (16 x `'b'`). SWD counters during the same runs showed `gcs_serial_ctrl_dbg_reply_dev103` and `gcs_serial_ctrl_dbg_reply_dev104` advancing, `gcs_serial_ctrl_dbg_last_available=16`, `gcs_serial_ctrl_dbg_last_reply_count=16`, plus active PIO TX/RX counters on both instances. A longer sequential validation pass produced near-perfect exact-match results (`SERIAL3 19/20`, `SERIAL4 19/20`; one-off misses appear to be harness timing/reply-latency artifacts rather than byte corruption). Conclusion: the PIO UART TX/RX path is now end-to-end functional on hardware for SERIAL3 and SERIAL4.

**current hardware status (updated 2026-03-29, session 6) — PIOUART BRING-UP PARTIAL (TX VERIFIED, RX INTEGRITY STILL OPEN):** SWD/GDB validation confirmed previous PIO IRQ storm lockup is fixed after correcting RP2350 PIO `FSTAT` bit mapping and tightening IRQ enable masks (only selected RXNEMPTY source enabled; stale force bits cleared). Both PIOUART instances are constructed and active (`serial3Driver`/`serial4Driver`), both receive SERIAL_CONTROL traffic (`dev103/dev104` counters advance), and both TX write counters increment on hardware. Loopback testing with GP14<->GP17 and GP19<->GP20 still shows non-deterministic/corrupted RX payloads (device 103 partial matches, device 104 mostly bad), so full bidirectional validation is not complete yet. Latest software changes in `PIOUART.cpp/.h`: fixed RP2350 `FSTAT` bit positions, RX debug counters, RX pin mode cleanup (no open-drain), and idempotent `_begin()` behavior for repeated SERIAL_CONTROL opens.

**current hardware status (updated 2026-03-30, session 8) — PIOUART STILL PARTIAL, RX NOW CLEAN-EMPTY INSTEAD OF CORRUPT:** Removed temporary TX signature injection from `PIORXDriver::_begin()` (that debug burst could contaminate loopback captures) and added explicit RX FIFO/ring purge at start of each PIOUART session. Reverted RX timing experiment back to midpoint sampling (`set x,7 [10]`). On hardware, blocking SERIAL_CONTROL token tests now return no payload (`dev103 rx_len=0`, `dev104 rx_len=0`) instead of prior garbage streams. Live SWD counters after these tests: `begin_count[0]=2`, `begin_count[1]=2`, TX active (`write_calls[0]=1504`, `write_calls[1]=14`, `write_bytes[0]=5901`, `write_bytes[1]=41`) but RX inactive (`rx_service_calls[0]=0`, `rx_service_calls[1]=0`, `rx_bytes[0]=0`, `rx_bytes[1]=0`). Register snapshot confirms pinmux/IRQ setup is correct (`GPIO14/17/19/20 FUNCSEL=6`, `PIO0_IRQ0_INTE=0x2`, `PIO0_IRQ1_INTE=0x8`) and RX FIFOs remain empty (`PIO0_FSTAT=0x0f000f00`, IRQ INTS clear). Current interpretation: software path is cleaner, but end-to-end RX still lacks proof with the present hardware loopback state; further wiring/signal-level verification is required before marking SERIAL3/SERIAL4 complete.

**current hardware status (updated 2026-03-30, session 9) — RX POLLING FALLBACK TESTED, FIFO STILL SEES NO BYTES:** `PIOUART.cpp` was switched from IRQ-owned RX draining back to a single-producer thread-context polling model (`_available()`, `_read()`, `_discard_input()` now call `_service_rx_fifo()`, and `_begin()` no longer enables RX IRQ). This removes the non-working IRQ dependency and preserves race-free ownership. Hardware result did not change: blocking loopback tests on `/dev/ttyACM1` still returned `rx_len=0` for both `device=103` and `device=104`. Live counters after the polling build showed the polling path is executing on SERIAL3 (`rx_service_calls[0]=25` in one snapshot, later >1200 after repeated tests) while `rx_bytes[0]` remained zero, so the software reader is running but the PIO RX FIFO is still empty. Additional raw GPIO/PIO snapshot during forced SERIAL3 transmission showed `DBG_PADOE=0x00084000` (PIO driving GP14 and GP19) and SIO `GPIO_IN` bits for GP14/GP17 were simultaneously low during a low-heavy TX stream, suggesting the external loopback path may exist electrically, but the RX state machine still is not producing bytes into FIFO. Current conclusion: TX-side software is alive and RX polling is alive, but end-to-end PIOUART RX/TX is still **not complete** on hardware; remaining work is now concentrated in live signal-level / PIO-state-machine debugging rather than the ring-buffer service path.

**current hardware status (updated 2026-03-30, session 7) — PIOUART RX STILL NOT END-TO-END CLEAN, BUT TWO REAL RX BUGS FIXED:** Additional hardware loopback on the latest build confirmed that PIOUART TX remains electrically good while RX is still not yet production-ready. Two genuine RX-side bugs were found and fixed in this session: (1) a software race in `PIOUART.cpp` where `_service_rx_fifo()` could run from both IRQ context and thread context, making `ByteBuffer` a multi-producer queue; RX draining is now IRQ-owned only; (2) the hardcoded RX PIO program incorrectly fell through to `push noblock` after a framing error (`wait 1 pin 0` then wrap-through), whereas the canonical Pico UART RX program jumps back to start and skips the push on bad stop bits. After fixing the framing path, the old repeated `0x0c` corruption pattern disappeared on hardware, proving that bug was real. A further hardware timing experiment advanced the first RX sample by one bit period (`set x,7 [2]` instead of `[10]`); with that change, PIOUART1 looped-back startup signature bytes changed from corrupted `0xD5` to clean `0xAA`, which is strong evidence that the RX sample phase was previously late. However, end-to-end payload echo is still not complete: latest isolated non-blocking tests on `/dev/ttyACM1` gave `dev103 ok 0 bad 42` and `dev104 ok 0 bad 34`, and clean blocking token tests still failed (`dev103` returned only `da00`, `dev104` no payload). Live counters after the timing test showed both ports still instantiated (`begin_count[0]=2`, `begin_count[1]=2`), but much heavier traffic on SERIAL3 than SERIAL4 (`write_calls[0]=3040`, `write_calls[1]=40`, `rx_bytes[0]=2057`, `rx_bytes[1]=27`). Current tree therefore contains real RX bring-up progress (race fix, framing-fallthrough fix, sample-phase shift, datasheet-style `RXF+3` byte read), but PIOUART RX/TX is **still partial** and requires more hardware debugging before marking SERIAL3/SERIAL4 verified.

**current hardware status (updated 2026-03-29, session 5) — BOOTLOADER --UPLOAD END-TO-END VERIFIED:** After session 4's STKOF/PSPLIM fix, `uploader.py` erase/program/verify/reboot completed 100% but the app CDC (`/dev/ttyACM1: Pico2`) never appeared — the board was stuck in `Pico2-BL`. Root cause: the new bare `NVIC_SystemReset()` path in `jump_to_app()` worked correctly for the first reset (XIP cache flushed by ROM re-entry), but then the fresh BL boot called `jump_to_app()` again (via `try_boot=true, timeout=0`), which hit the same `NVIC_SystemReset()` again → infinite reset loop. Diagnosed via SWD/GDB: `WATCHDOG->SCRATCH[1] = 0xB007CAFE` still set after multiple resets (never cleared). **Fix:** Two-phase flag state machine: `jump_to_app()` checks `SCRATCH[1]` — if it equals `0xB007CA11U` (phase 2, "XIP clean"), clear flag and fall through to bare `do_jump()`; otherwise set `0xB007CAFEU` and `NVIC_SystemReset()` (phase 1, "please reset"). `AP_Bootloader.cpp` main() advances to phase 2 by writing `0xB007CA11U` before setting `try_boot=true, timeout=0`. Result: phase-1 reset flushes XIP via ROM, phase-2 jump executes from clean hardware state. **Verified 2026-03-29:** `python3 uploader.py --port /dev/ttyACM1 arducopter.apj` → erase/program/verify/reboot 100% → `/dev/ttyACM1: Pico2` appears → 193 bytes MAVLink v2 (`fd09...`) confirmed. Commits: `bae099c` (NVIC_SystemReset phase-1), `af47ef0` (two-phase fix).

**current hardware status (updated 2026-03-29, session 4) — BOOTLOADER JUMP_TO_APP STKOF FIX:** After flashing app firmware via the bootloader, the app CDC (`/dev/ttyACM1: Pico2`) was not appearing — the board entered LOCKUP with CFSR=0x00108200 (STKOF UsageFault + PRECISERR), HFSR=0x40000000 (forced HardFault), PC=0xeffffffe. Root cause: Cortex-M33 PSPLIM/MSPLIM registers. ChibiOS bootloader sets PSPLIM = top of its own stack = 0x20000600. The app's initial SP (from vector table word 0) is also 0x20000600. When `do_jump()` executes `mov sp, r0` with r0 == PSPLIM, the Cortex-M33 STKOF check fires: new SP <= PSPLIM → UsageFault → HardFault → `save_fault_watchdog` loop → LOCKUP → app CDC never enumerates. Diagnosed via SWD/GDB: PSP exception frame's saved PC = `0x100007b2` (`do_jump()+6` = the `mov sp, r0` instruction), PSPLIM_S = 0x20000600 = app initial SP. **Fix:** `__set_MSPLIM(0); __set_PSPLIM(0)` (CMSIS intrinsics) inserted before `port_disable()` in `jump_to_app()` in `Tools/AP_Bootloader/bl_protocol.cpp`, inside `#if defined(RP2350)`. **Robustness:** Added compile-time ternary to handle two possible hwdef `APP_START_ADDRESS` variants: `((APP_START_ADDRESS & 0xFFU) == 0U) ? APP_START_ADDRESS + 0x80U : APP_START_ADDRESS` — if low byte is 0x00 (imagedef base, e.g. 0x10010000) add 0x80 to reach the ARM vector table; if low byte is 0x80 (e.g. 0x10010080) use as-is. VTOR now set from adjusted `app_base` instead of raw `APP_START_ADDRESS` to guarantee consistency. Current Pico2 hwdef-bl.dat already defines `APP_START_ADDRESS 0x10010080` so the ternary takes the identity path. **Verified 2026-03-29:** `/dev/ttyACM1: Pico2` appears after BL timeout, `c1_boot_stage=0x4D` (idle, both cores running). Commits: `54bb763` (soft-jump fix), `9fab42f` (PSPLIM/VTOR fix), `bb4faf5` (robust APP_START_ADDRESS detection).

**current hardware status (updated 2026-03-28, session 3) — BOOT HARDENING COMPLETE:** Two new defensive changes make the RP2350 self-recover from any soft-reset or spurious IRQ rather than hanging permanently. (1) `start_core1()` deadlock eliminated: `hal_lld.c` now PSM-resets core1 (`PSM->FRCE_OFF = PSM_ANY_PROC1`, wait DONE, release) before the 6-step ROM FIFO handshake. Root cause: a soft reset (watchdog or bootloader `jump_to_app()` which is a bare `BX` without SYSRESETREQ) left core1 executing `c1_main`, which drained the FIFO without echoing back — causing core0 to block forever in `fifoBlockingRead` at seq=0. Diagnosis: FIFO_ST=0x00000002 (VLD=0, only RDY set), `c1_boot_stage=0x4D` proving core1 was running. PSM reset is <1 µs and idempotent. (2) `_unhandled_exception` override: `board.c` overrides the ChibiOS weak symbol (default: infinite loop `.stay: b .stay`) with a strong `NVIC_SystemReset()` call, guarded by `defined(PICO2_AVAILABLE)`. Any spurious IRQ that reaches the default handler (e.g. an IRQ fired by floating GPIO lines while its ChibiOS vector was not registered) now causes a clean chip-level self-reset in <1 µs rather than a silent permanent hang. Together these two changes mean the board is always self-recovering: reset clears the IRQ condition, PSM guarantees core1 is back in ROM wait, and start_core1 completes the handshake cleanly. Verified: USB CDC on `ttyACM1`, `c1_boot_stage=0x4D` (idle), `hw_check_gdb.py` 21/21 PASS (2026-03-28). Also on this session: udev rule `/etc/udev/rules.d/60-rp2350-bootsel.rules` added for USB ID `2e8a:000f` (RP2350 BOOTSEL mode) on rebuilt laptop (picotool needs unprivileged access), and `libraries/GCS_MAVLink/include` symlink re-created pointing to `build/Pico2/libraries/GCS_MAVLink/include` (symlink was absent on reinstalled laptop, causing build failure `version.h: No such file or directory`).

**current hardware status (updated 2026-03) - ROOT CAUSE FOUND AND FIXED: The firmware was crashing on every boot due to a spurious I2C1 interrupt (IRQ37/exception53). Diagnosis via SWD/OpenOCD showed PC always stuck at `_unhandled_exception` (`0x1001127a`) with xPSR exception=53. The RP2350's GPIO15 (I2C1_SCL) and GPIO18 (I2C1_SDA) are floating on the test carrier board, generating spurious I2C bus activity → I2C1 IRQ fires → `VectorD4` handler (`hal_i2c_lld.c:375`) runs during idle → ChibiOS `__port_irq_epilogue` performs a corrupted context switch → CPU jumps to `DebugMon_Handler` → `_unhandled_exception` infinite loop. Fix: commented out `I2C_ORDER I2C1` and `PA15/PA18 I2C1_SCL/SDA` pin definitions in `hwdef.dat`. Without `I2C_ORDER`, the hwdef preprocessor does not generate `HAL_I2C_DEVICE_LIST`, so `i2cStart()` is never called for I2C1, NVIC IRQ37 stays disabled, and spurious I2C bus activity cannot crash the firmware. Re-enable I2C by restoring those 3 lines once carrier board has pull-up resistors or I2C devices connected. RESULT: USB CDC now enumerates (`usb_event_configured_count=1`), `/dev/ttyACM1` is stable, ArduCopter running in thread mode. Added USB event counters (reset/config/suspend/unconfig + last event time) in `usbcfg.c`/`usbcfg.h`/`system.cpp` for future USB stability monitoring.

**USB CDC Bootloader — FIXED (2025-07):**
Root cause identified and fixed: `vcom_strings[1..3]` were `{0, NULL}` because `setup_usb_strings()` was never called. The Pico2 BL uses `rp2350_imagedef_ref.c` for `__late_init()` (overrides `board.c`), which was missing the `setup_usb_strings()` call. Without valid string descriptors, Linux USB retried with 5-second timeouts × 3 = 15s enumeration delay, causing the BL to time out before USB was usable. Fix: added `setup_usb_strings()` call in `Tools/AP_Bootloader/rp2350_imagedef_ref.c` `__late_init()`. USB now enumerates in ~3 seconds as "ArduPilot Pico2-BL". `uploader.py --port /dev/ttyACM1` protocol works: INSYNC+OK received, board info read, erase started successfully.

**ChibiOS SMP exploration (2026-03, abandoned):** ChibiOS full-SMP (CH_CFG_SMP_MODE=TRUE) was explored but abandoned due to unacceptable spinlock overhead. Exploration artifacts (SMP_REG_LOCK, spinlock release at startup, stage canary writes) are committed to the ChibiOS submodule, guarded by `#if CH_CFG_SMP_MODE == TRUE`, zero cost in production. Current architecture: CH_CFG_SMP_MODE=FALSE (single ChibiOS instance on core0) + RP_CORE1_START=TRUE (bare-metal c1_main.c WFE dispatcher on core1). Rate controller offloaded to core1 via c1_run_sync() in rate_thread.cpp.
